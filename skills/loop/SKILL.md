# Loop — 범용 자율 루프 프로토콜

## 0. 진입 판단 (Entry-Level Routing)

태스크를 받으면 **(1) 도메인 판단(섹션 4)** → **(2) 복잡도 판단** 순서로 진행한다.

| 신호 | L0 직행 | 전체 루프 진입 |
|------|---------|---------------|
| 수정 파일 수 | 1개 | 다수 |
| 반복 필요성 | 1회로 완료 가능 | 반복 예상 |
| 도메인 전환 가능성 | 없음 | 있음 |

**L0 직행 시**: 기획 질문, 정의서, 루프 기록을 생략한다. core.md + 도메인 규율을 적용하여 실행 → 검증 → 완료의 3단계로 처리한다. 실행 중 에스컬레이션 트리거가 발동하면 전체 루프로 승격한다.

**전체 루프 진입 시**: 아래 섹션 1부터 진행한다.


## 1. 기획 질문 루프

루프 정의서가 없는 상태에서 태스크를 받으면, 아래 질문으로 정의서를 구성한다.
한 번에 모든 질문을 던지지 말고, 대화 흐름에 맞춰 2-3개씩 묶어서 질문한다.

### 필수 질문 7가지

| 순서 | 질문 | 수집 대상 |
|------|------|-----------|
| 1 | 이 작업의 최종 목표가 뭔가요? | `loop.name` + 목표 |
| 2 | 어떤 파일/범위를 수정하나요? | `loop.modify` |
| 3 | 수정 후 어떻게 실행하나요? (명령어, 수동 검토 등) | `loop.run` |
| 4 | 결과를 뭘로 평가하나요? (메트릭, 기준, 통과 조건) | `loop.evaluate` |
| 5 | keep/discard/refine 판단 기준은? | `loop.judge` |
| 6 | 제약조건이 있나요? (수정 금지 파일, 시간 제한 등) | `loop.constraints` |
| 6b | 수정하면 안 되는 부분이 있나요? (글쓰기/퇴고 등 주관적 평가 도메인에서) | `loop.constraints` (보존 영역) |
| 7 | 언제 루프를 끝내나요? | `loop.exit` |

### 질문 생략 규칙

- 사용자가 태스크 설명에서 이미 답한 항목은 묻지 않고 채운다.
- 대화 컨텍스트에서 명확히 추론 가능한 항목도 묻지 않는다.
- 최소한 `modify`, `run`, `evaluate`는 반드시 확보한다.

### 정의서 확인

질문 완료 후 루프 정의서 YAML을 생성하여 보여주고, 사용자에게 확인받는다.
사용자가 "좋아", "ㄱ", "시작" 등 긍정 응답을 하면 루프를 시작한다.
수정 요청이 오면 해당 부분만 고치고 다시 확인받는다.

L0 단순 태스크(질문 루프에서 충분히 합의된 경우)에서는 YAML 전문 대신 1줄 요약으로 축약할 수 있다: "이렇게 진행합니다 — [요약]. 이의 있으면 말씀해주세요."


## 2. 루프 프로토콜

```
[기획] → [정의] → [실행] → [평가] → [판단] → [기록] → [반복/종료]
```

### [기획] Planning

- 루프 정의서가 없으면 → 섹션 1의 기획 질문 루프 실행
- 루프 정의서가 있으면 → 바로 [정의]로 진행
- 마일스톤 규모의 복잡한 기획이면 → System 2 에스컬레이션

#### 브레인스토밍 (기획 심화)

마일스톤 미만의 설계/전략 판단이 필요한 기획에서는 처음 떠오른 접근을 그대로 채택하지 않는다. 처음 생각은 누구나 하는 생각이다. 마일스톤 규모면 System 2(L3) 에스컬레이션 후 워커가 브레인스토밍을 수행한다.

- **사색**: 워커가 초기 접근을 독립적으로 변주한다. 다른 관점, 다른 제약, 다른 우선순위로 같은 문제를 재해석.
- **토론**: 변주된 접근들을 교차 비교하여 공통점과 차이점을 정리하고, 사용자에게 선택지를 제시.
- 루프가 목적에서 이탈하지 않도록, 변주마다 "원래 목표에 대한 답인가?"를 체크한다.
- 사색으로 소화하고, 토론으로 통합한다.

### [정의] Definition

- 루프 정의서 YAML 확정
- 이번 반복에서 구체적으로 무엇을 수정할지 결정
- 첫 반복이면 베이스라인 상태 확인 (현재 테스트 결과, 현재 메트릭 등)

### [실행] Execution

- 수정 전에 대상 파일의 git 추적 여부를 확인한다
  - git 추적 파일: discard 시 `git restore`로 복원 가능 (기본 동작)
  - git 미추적 파일: 수정 전 원문을 메모리에 보관한다. discard 시 보관본으로 복원. keep 시 보관본 폐기.
- `loop.modify`에 지정된 범위 내에서만 수정
- `loop.constraints`를 위반하지 않는지 수정 전에 확인
- 수정 완료 후 `loop.run`에 지정된 명령 실행
- 실행 결과(stdout, stderr, 로그)를 수집

### [평가] Evaluation

- `loop.evaluate`에 정의된 메트릭/기준으로 결과 측정
- 이전 반복 결과와 비교 (첫 반복이면 베이스라인과 비교)
- 평가 결과를 수치/근거와 함께 명시적으로 제시

### [판단] Judgment — 3단 판단

결과를 아래 3가지 중 하나로 판단한다. **판단 근거를 반드시 명시한다.**

| 판단 | 조건 | Git 동작 |
|------|------|----------|
| **keep** | `loop.judge.keep` 조건 충족 | 자체 리뷰 → `git add + commit` (변경 확정) |
| **discard** | `loop.judge.discard` 조건 충족 | `git checkout/restore` (변경 폐기) |
| **refine** | 부분적 개선, 부분적 악화 | 좋은 부분 유지 + 나쁜 부분 수정 후 재실행 |

keep 판단 시 커밋 전에 자체 리뷰를 수행한다 (dev 도메인: 변경된 부분의 보안/에러 핸들링/엣지 케이스 점검). 리뷰에서 이슈 발견 시 refine으로 전환. 이후 guard의 완료 전 재검토(전체 흐름 영향 확인)를 거쳐 커밋. 순서: loop 자체 리뷰 → guard 재검토 → 커밋.

#### 보존 영역 갱신 (주관적 평가 도메인)

refine/discard 판단 시, "이전 버전이 더 나았다"고 판단된 영역이 있으면 `loop.constraints`의 보존 영역에 추가한다. 보존 영역은 사전 1회 고정이 아니라 반복마다 갱신될 수 있다. 갱신 시 사용자에게 확인받는다.

#### 판단 시 에스컬레이션 체크

매 판단에서 FOR(Feeling of Rightness)을 자가 점검한다 (트리거 4가지와 레벨 매핑은 섹션 7 참조). 하나라도 해당하면 에스컬레이션 레벨을 올린다.

### [기록] Record

- keep 판단 시: 커밋 메시지에 반복 번호 + 메트릭 변화 포함
- discard 판단 시: 왜 폐기했는지 간단히 기록 (다음 반복에서 같은 실수 방지)
- refine 판단 시: 무엇을 유지하고 무엇을 수정하는지 기록

### [반복/종료] Loop/Exit

- `loop.exit` 조건 충족 → 루프 종료, 최종 결과 요약
- 조건 미충족 → [정의] 단계로 돌아가 다음 반복 시작
- 종료 시 보고: 총 반복 횟수, 메트릭 변화 추이, 최종 상태
- exit 조건이 "N회 연속 개선폭 < X%" 형식인 경우: refine 반복은 카운터를 리셋하지 않는다 (refine은 미완성 개선이므로 수렴 판단에 포함)
- 루프 종료 시 후속 루프에 넘길 컨텍스트가 있으면 **Loop Artifact**로 기록한다:
  - 산출물 경로 (생성/수정된 파일)
  - 핵심 결과 (데이터 스키마, 분석 결론, 코드 구조 등)
  - 후속 루프에 전달할 제약/전제 조건
  - 규율 전이: 이전 루프에서 발견된 판단 중 후속 루프에 영향을 주는 것 (예: "이 데이터는 이상치가 상위 5%에 집중 → 자동화 시 이상치 필터링 필수")
  - Loop Artifact가 3건 이상 누적되면, 핵심 항목만 유지하고 나머지는 요약 병합한다
  - 이 기록은 도메인 전환 시 새 루프 정의서의 컨텍스트로 활용된다

#### 복기 (루프 종료 후)

3회 이상 반복한 루프가 종료되면, 루프 리포트 작성 전에 복기를 수행한다.

- **착수 순서**: 각 반복에서 선택한 접근 순서가 최선이었는가? 다른 순서였으면 반복 횟수가 줄었을까?
- **의도 vs 결과**: 각 반복에서 의도한 것과 실제 결과의 차이를 검토. discard/refine이 발생한 반복에서 의도가 어디서 틀어졌는가?
- **다른 가능성**: 시도하지 않은 접근 중 더 나았을 가능성이 있는 것을 메모. 다음 루프에서 참고.
- 복기 결과는 루프 리포트에 1~2줄로 요약하여 포함한다.
- 도메인 전환이 2회 이상인 복합 태스크에서는 **크로스 도메인 복기**도 수행한다: 이전 도메인에서의 판단이 후속 도메인에 어떤 영향을 미쳤는지 검토.


## 3. 루프 정의서 템플릿

### 개발 루프

```yaml
loop:
  name: "기능 구현/버그 수정"
  modify: "src/ 내 관련 파일"
  run: "uv run pytest tests/ -v"
  evaluate:
    metrics: ["test_pass_rate"]
    source: "pytest 출력"
  judge:
    keep: "모든 테스트 통과"
    discard: "기존 테스트 깨짐"
    refine: "새 테스트 통과, 기존 일부 실패"
  constraints:
    - "공개 API 시그니처 변경 금지"
  exit: "모든 테스트 통과 + 자체 코드리뷰 완료"
```

### ML 루프

```yaml
loop:
  name: "모델 성능 개선"
  modify: "train.py — 아키텍처, 하이퍼파라미터"
  run: "uv run train.py > run.log 2>&1"
  evaluate:
    metrics: ["macro_f1", "accuracy"]
    source: "grep 'macro_f1' run.log"
  judge:
    keep: "macro_f1 개선"
    discard: "macro_f1 동일 또는 하락"
    refine: "일부 클래스만 개선, 다른 클래스 하락"
  constraints:
    - "prepare.py 수정 금지"
    - "학습 시간 5분 이내"
  exit: "3회 연속 개선폭 < 0.5%"
```

### 글쓰기 루프

```yaml
loop:
  name: "퇴고"
  modify: "대상 문서 파일"
  run: "퇴고 1회 수행 — 수정 전후를 diff로 제시"
  evaluate:
    criteria: ["문체 일관성", "흐름", "군더더기 제거"]
    require_user_confirmation: true  # 주관적 평가 → 자동 keep 방지
  judge:
    keep: "이전 버전 대비 개선"
    discard: "개악 — 원래 표현이 더 나음"
    refine: "일부 개선, 일부 과도한 수정"
  exit: "수정 사항 5개 미만"
```

`require_user_confirmation: true`가 설정되면, keep/discard 판단 전에 반드시 사용자에게 수정안을 제시하고 확인을 받는다. 코드/ML처럼 정량 메트릭으로 자동 판단할 수 없는 도메인(글쓰기, 디자인 등)에서 사용.


## 4. 도메인 판단 (대화 기반, 2차)

session-start 훅이 파일 기반 1차 판단을 수행한다.
이 스킬은 태스크 수신 시 대화 컨텍스트 기반 **2차 판단**을 수행한다.

### 입력 신호 → 도메인 매핑

| 신호 (키워드/맥락) | 도메인 | 규율 파일 |
|---------------------|--------|-----------|
| 테스트, 리팩토링, API, 배포, 커밋, PR | `dev` | `dev.md` |
| 학습, F1, loss, 에폭, GPU, 모델, 파인튜닝 | `ai-ml-dl` | `ai-ml-dl.md` |
| 데이터, 시각화, 분석, 전처리, 통계, 피벗 | `analysis` | `analysis.md` |
| 문서, 정리, 목차, 용어, 매뉴얼 | `document` | `document.md` |
| 위에 해당 없음 | `general` | `general.md` |

### 판단 규칙

1. 태스크 지시에서 키워드/맥락을 확인하여 도메인 결정
2. 한 태스크에 여러 도메인 신호가 섞이면, **주 작업** 기준으로 1개 선택
3. 판단 불가 시 `general`로 시작하되, "이 작업 어떤 도메인에 가까운가요?"라고 한 번 질문
4. 세션 내에서 태스크가 바뀌면 도메인도 재판단

### 사용자 오버라이드

사용자가 "이건 분석 작업이야", "ML로 봐줘" 등 자연어로 도메인을 지정하면 즉시 전환한다.
프로젝트 GEMINI.md의 `## sonmat` 섹션에 `domain: ai-ml-dl` 같은 고정 설정도 존재할 수 있다.
고정 설정이 있으면 해당 도메인을 기본으로 사용하되, 자연어 오버라이드가 우선한다.

### 도메인 전환 절차

세션 중 태스크가 바뀌어 도메인이 변경될 때 아래 절차를 따른다.

1. **현재 루프 종료**: 진행 중인 루프가 있으면 종료하고 루프 리포트를 출력한다. 종료할 수 없으면 사용자에게 확인한다.
2. **이전 도메인 규율 비활성화**: 현재 도메인 규율 파일을 내린다. core.md는 유지.
3. **새 도메인 판단 + 규율 활성화**: 섹션 4의 매핑표로 새 도메인을 판단하고, 해당 규율 파일을 로드한다.
4. **오버라이드 재확인**: 프로젝트 GEMINI.md의 `## sonmat` 섹션을 다시 확인하여 새 도메인에 맞는 오버라이드를 적용한다.
5. **사용자 알림**: 전환 사실, 이전/이후 도메인, 새 규율의 핵심 포인트를 간결하게 안내한다.

전환 시 이전 루프의 Loop Artifact(섹션 2 [반복/종료] 참조)를 새 루프 정의서의 컨텍스트로 전달한다. 규율 전이 항목은 후속 루프의 `constraints`에 추가하거나, 반복 발견 시 `discipline.add` 오버라이드로 영구화한다.


## 5. 규율 오버라이드 파싱

프로젝트 GEMINI.md에서 `## sonmat` 또는 `## sonmat 오버라이드` 섹션을 읽는다.

### 파싱 대상 형식

```markdown
## sonmat
domain: ai-ml-dl
discipline:
  disable:
    - "dev.md > TDD"
  add:
    - "커밋 전 ruff format 필수"
```

### 처리 규칙

1. `domain`: 1차/2차 판단 결과를 덮어쓰는 고정 도메인
2. `discipline.disable`: 해당 규율 파일에서 지정된 섹션을 비활성화
   - 형식: `"파일명 > 섹션명"` (예: `"dev.md > TDD"`)
   - 비활성화된 규율은 워커 프롬프트에 주입하지 않는다
3. `discipline.add`: 추가 규율을 도메인 규율 뒤에 붙인다
   - 자유 형식 텍스트, 한 줄씩

### 오버라이드 없는 경우

`## sonmat` 섹션이 없으면 기본 동작:
- 도메인: 1차/2차 판단 결과 사용
- 규율: `core.md` + 도메인 규율 파일 전체 적용
- 추가 규율 없음


## 6. 규율 주입 절차

도메인 판단 + 오버라이드 파싱이 완료되면, 아래 순서로 규율을 적용한다.

1. `core.md` 로드 (항상)
2. 도메인 규율 파일 로드 (판단된 도메인에 해당하는 파일)
3. `discipline.disable`에 해당하는 섹션 제거
4. `discipline.add`에 해당하는 항목 추가
5. 결과물을 System 1 처리에 적용하거나, System 2 워커 스폰 시 프롬프트에 포함

System 2 워커 스폰 시 프롬프트 구성:
```
[역할 지시] (1줄)
[규율 텍스트] (core.md + 도메인 규율, 오버라이드 적용 후)
[루프 컨텍스트] (루프 정의서 + 현재 반복 상태)
[태스크 지시] (구체적 작업 내용)
```


## 7. 에스컬레이션 시스템

Kahneman의 이중 프로세스 이론 + Evans & Stanovich(2013) Default-Interventionist 모델 기반.
System 1이 기본 처리하되, FOR(Feeling of Rightness)이 낮아지면 System 2로 에스컬레이션한다.

### 판단 방식

에스컬레이션 판단은 **프롬프트 기반 FOR 체크**로 수행한다. 훅 기반이 아니다.

- 훅은 도구 호출의 입출력만 보고, 대화 컨텍스트를 모른다
- 에스컬레이션은 "이 상황이 얼마나 복잡한지" 맥락 이해가 필요하다
- 따라서 매 판단 단계에서 아래 4가지 트리거를 자가 점검한다

### 트리거 4가지

| 트리거 | 감지 조건 | 예시 |
|--------|-----------|------|
| **Surprise** | 예상 결과 ≠ 실제 결과 | 테스트 통과 예상인데 실패, 메트릭 역행 |
| **Error** | 같은 실패 2회 이상, 수정이 다른 곳을 깨뜨림 | 같은 테스트 2번 실패, fix → 다른 테스트 깨짐 |
| **Fluency break** | 참조 불가, 의존성 누락, 스코프 불명확 | 파일/경로 못 찾음, import 누락, 범위 모호 |
| **Conflict** | 현재 작업이 규율/기존 계획과 충돌 | 규율 위반 감지, 스코프 변경 vs 기존 계획 |

### 레벨 4단계

| 레벨 | 이름 | 동작 | 예시 |
|------|------|------|------|
| **L0** | System 1 처리 | 스킬로 바로 실행, 에스컬레이션 없음 | 오타 수정, 단일 파일 포맷, 간단한 구현 |
| **L1** | System 1 강화 | 멈추고 한 번 더 확인 — 영향 범위 체크 | 여러 파일 수정 전 의존관계 확인, 테스트 1회 실패 후 원인 분석 |
| **L2** | System 2 스폰 | sonmat-worker 에이전트 1개 스폰 (규율 주입 + 깊은 분석) | 테스트 2회 실패, 메트릭 역행, 복잡한 디버깅 |
| **L3** | System 2 확장 | 복수 워커 병렬 스폰 (마일스톤 단위) | 아키텍처 변경, 대규모 리팩토링, 병렬 실험 |

### 트리거 → 레벨 매핑

| 상황 | 트리거 | 레벨 |
|------|--------|------|
| 첫 번째 예상 외 결과 | Surprise | L0 → L1 |
| 같은 실패 2회 반복 | Error | L1 → L2 |
| 수정이 다른 곳을 깨뜨림 | Error + Surprise | → L2 |
| 참조 파일/경로 못 찾음 | Fluency break | L0 → L1 (탐색 후 재시도) |
| 의존성 누락 + 스코프 불명확 | Fluency break | → L2 (분석 필요) |
| 규율 위반 감지 | Conflict | → L1 (확인 후 판단) |
| 스코프 변경 vs 기존 계획 충돌 | Conflict | → L2 (재기획 필요) |
| 마일스톤 규모 기획 필요 | 복합 | → L3 |
| L2 워커가 해결 못 함 | Error 누적 | → L3 (병렬 접근) |

### 에스컬레이션 시 사용자 안내

에스컬레이션 발생 시 반드시:
1. 어떤 트리거가 발동했는지 명시한다
2. 어떤 레벨로 올리는지 명시한다
3. 해당 에스컬레이션을 방지할 수 있는 루프 정의서 커스텀 방법을 제안한다 (강제 아님)

에스컬레이션이 해소되면 복귀를 알린다:
- 형식: `✓ [트리거] 해소 → L[N] 복귀`
- 예시: `✓ Surprise 해소 → L0 복귀 (경계값 수정으로 테스트 통과)`


## 8. 워커 디스패치 프로토콜

L2/L3 에스컬레이션 시 sonmat-worker를 스폰하는 절차.

### 프롬프트 구성 (5파트)

워커 프롬프트는 아래 5개 파트를 순서대로 조합한다. 총 2,000-5,000 토큰을 목표로 한다.

```
[1. 역할 지시]    — 1줄. "디버거/실행자/리서처/리뷰어 역할을 수행하라"
[2. 규율 텍스트]  — core.md + 도메인 규율 (오버라이드 적용 후). ~80줄. 단일 태스크(디버깅 1건 등)에서는 핵심 원칙만 ~20줄로 축약 가능.
[3. 루프 컨텍스트] — 루프 정의서 YAML + 현재 반복 번호/상태. 해당 시에만 포함.
[4. 프로젝트 맵]  — 주요 경로, 참조 파일 위치. 해당 시에만 포함. ~10줄.
[5. 태스크 지시]  — 구체적 작업 내용. 무엇을 하고, 결과를 어떻게 보고할지.
```

### 디스패치 형식

L2 에스컬레이션 시 아래 형식으로 Agent 도구를 호출한다:

```
Agent 도구 호출:
  prompt: |
    [역할] 디버거 역할을 수행하라.

    [규율]
    {core.md 내용}
    {도메인 규율 내용 — 오버라이드 적용 후}

    [루프 컨텍스트]
    {루프 정의서 YAML}
    현재 반복: #N, 이전 판단: {keep/discard/refine}, 트리거: {발동된 트리거}

    [프로젝트 맵]
    - src/: 소스 코드
    - tests/: 테스트
    - {기타 주요 경로}

    [태스크]
    {구체적 작업 지시}
    완료 후 결과를 아래 형식으로 보고하라:
    - 수행한 작업
    - 변경한 파일
    - 검증 결과
    - 추가 조치 필요 여부
```

### 결과 처리

워커가 반환한 결과를 아래 순서로 처리한다:

1. **결과 수신**: 워커의 보고를 읽는다
2. **검증**: 워커가 수행한 변경이 루프 정의서의 제약조건을 위반하지 않는지 확인한다
3. **루프 복귀**: 결과를 루프의 [평가] 단계로 전달하여 정상 루프를 계속한다
4. **사용자 보고**: 에스컬레이션이 발생했던 사실, 워커가 수행한 작업, 결과를 요약하여 보고한다

### 워커 상태 코드 → 루프 처리

워커가 반환하는 상태 코드에 따라 루프 처리를 분기한다.

| 상태 코드 | 루프 처리 |
|-----------|-----------|
| `DONE` | 결과를 [평가] 단계로 전달. 정상 루프 계속. |
| `DONE_WITH_CONCERNS` | 결과를 [평가]로 전달하되, 우려 사항을 사용자에게 함께 보고. |
| `NEEDS_CONTEXT` | 워커에 추가 컨텍스트 제공 후 재실행, 또는 사용자에게 정보 요청. |
| `BLOCKED` | 워커 실패로 간주. L3 에스컬레이션(병렬 접근) 또는 사용자에게 판단 위임. |

워커가 `loop.modify` 범위 외 파일 변경이 필요하다고 판단한 경우, 직접 변경하지 말고 필요성과 이유를 보고에 포함해야 한다. 메인 세션에서 사용자 확인 후 범위를 확장한다.

### L3 병렬 디스패치

L3 에스컬레이션 시 복수 워커를 병렬로 스폰한다.

**태스크 분할 기준:**
- 파일 의존성이 없는 독립 태스크로 분할한다
- 같은 파일을 수정하는 태스크는 같은 워커에 묶는다
- 각 워커에 동일한 규율 텍스트를 주입하되, 태스크 지시만 다르게 한다

**결과 병합:**
1. 모든 워커 완료를 대기한다
2. 각 워커 결과를 수집한다
3. 충돌 여부를 확인한다 (같은 파일을 다른 워커가 수정한 경우)
4. 충돌 없으면 결과를 병합하고 루프 [평가]로 전달한다
5. 충돌 있으면 사용자에게 보고하고 판단을 요청한다

**병렬 스폰 예시:**
```
워커 A: "모듈 X 리팩토링" — modify: src/module_x/
워커 B: "모듈 Y 리팩토링" — modify: src/module_y/
워커 C: "통합 테스트 작성" — modify: tests/integration/
```

각 워커는 독립적으로 실행되며, 완료 후 메인 세션에서 결과를 통합한다.

**역할 유동 배분:**

병렬 워커 운영 시, 워커 간 기여도에 편차가 생긴다. 핵심 작업을 수행하는 워커를 파악하여 나머지가 서포트하도록 유동적으로 역할을 조정한다.

- 메인 세션이 조율자 역할을 한다. 워커 진행 중에는 진행 상태 파악 + 태스크 재배분, 워커 완료 후에는 결과 병합→평가→판단→기록.
- 워커로부터 바텀업 피드백을 받는다: 워커가 태스크 지시의 문제를 발견하면 보고하고, 메인 세션이 이를 수용하여 지시를 수정할 수 있다.
- 관찰자의 판단도 틀릴 수 있다. 워커의 현장 판단이 더 정확하면 그쪽을 따른다.


## 9. 사용자 성장 구조

철학: "세상에 나쁜 사용자는 없다" — 루프의 모든 판단과 에스컬레이션을 투명하게 보여줌으로써 사용자가 시스템을 이해하고 성장할 수 있도록 한다.

### 3원칙

1. **열린 커스텀 표면**: 루프 정의서, 규율 오버라이드, 프로젝트 맵은 언제든 사용자가 수정할 수 있다. 막내는 이 표면을 적극적으로 안내한다.
2. **적시 안내**: 에스컬레이션이 발생하는 순간, 해당 상황을 방지하거나 개선할 수 있는 커스텀 방법을 제안한다. 강제하지 않는다.
3. **투명한 판단**: keep/discard/refine 판단과 모든 에스컬레이션에 대해 "왜"를 반드시 명시한다. 사용자가 루프의 논리를 이해할 수 있어야 한다.

### 에스컬레이션 투명성 출력 형식

에스컬레이션 발생 시 아래 형식으로 즉시 출력한다:

```
⚡ [트리거 종류] 감지
  원인: [구체적 원인]
  조치: L[N] → [수행할 조치]
  💡 팁: [커스텀 가능한 개선 제안] (해당 시에만)
```

예시:
```
⚡ Fluency break 감지
  원인: src/utils/parser.py 경로를 찾을 수 없음
  조치: L0 → L1 (경로 탐색 후 재시도)
  💡 팁: 프로젝트 맵에 주요 경로를 등록하면 이런 중단을 줄일 수 있어요.
```

```
⚡ Error 감지
  원인: test_tokenizer.py 동일 실패 2회 반복
  조치: L1 → L2 (디버거 워커 스폰)
```

### 루프 리포트 형식

루프 종료 시 아래 형식으로 최종 보고한다:

```
📊 루프 리포트
  총 N회 반복 | keep M | discard K | refine J
  효율: keep/(전체) = X% [discard 50% 이상이면 기획 재검토 권장]
  에스컬레이션: [발생 내역 — 없으면 "없음"]
  복기: [3회+ 반복 시 — 착수 순서/의도vs결과/대안 1~2줄 요약]
  결과: [최종 상태 요약]
```

1회 keep으로 종료된 간단한 루프는 간소화 형식을 사용할 수 있다:
`📊 1회 완료. [결과 요약]. 에스컬레이션 없음.`

예시:
```
📊 루프 리포트
  총 5회 반복 | keep 3 | discard 1 | refine 1
  에스컬레이션: Error(2회차, L2 스폰) → 해결
  결과: macro_f1 0.847 → 0.891 (+0.044), 모든 테스트 통과
```

도메인 전환이 포함된 복합 태스크에서는 통합 리포트를 작성한다:

```
📊 전체 태스크 리포트
  도메인 전환: N회 ([도메인 목록])
  루프 요약:
    [도메인1] N회 | keep/discard/refine | 에스컬레이션
    [도메인2] ...
  Loop Artifact: N건 연쇄
  크로스 도메인 복기: [교훈]
  미완료: [항목]
```

### 커스텀 제안 트리거

아래 상황에서 사용자에게 커스텀 제안을 한다. 모두 제안이며 강제가 아니다.

| 상황 | 제안 내용 |
|------|-----------|
| Fluency break — 경로 못 찾음 | 프로젝트 맵(워커 프롬프트 파트 4)에 해당 경로 추가 제안 |
| 같은 오버라이드가 반복 적용됨 | GEMINI.md의 `## sonmat` 섹션에 영구 등록 제안 |
| 같은 루프 정의서를 재사용 | 루프 정의서 템플릿으로 저장 제안 (섹션 3 참조) |
| 에스컬레이션이 특정 패턴으로 반복됨 | 루프 정의서의 `constraints` 또는 `judge` 기준 조정 제안 |
| 사용자가 불편/개선 요청을 했고 sonmat이 놓친 문제인 경우 | GitHub 이슈 생성을 제안 (사소한 것도 가능, 체감 경험 개선이 목표) |
| 권한(allow) 거부로 작업이 중단됨 | 해당 권한을 settings.json에 추가하는 방법을 안내. 첫 세팅이 아니라 필요한 시점에 안내 |

### 제안 출력 형식

커스텀 제안은 에스컬레이션 출력의 `💡 팁:` 항목 또는 루프 리포트 하단에 덧붙인다.

```
💡 이 제안을 적용하려면:
  [구체적 수정 방법 — 어떤 파일의 어느 섹션을 어떻게]
  적용하지 않아도 루프는 정상 동작합니다.
```
